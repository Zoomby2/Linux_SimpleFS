# README для модуля ядра SimpleFS

## Задание

Необходимо реализовать собственную ФС на базе существующего блочного устройства. ФС должна содержать 2 копии superblock (в начале диска и с определенным смещением в N секторов). Каждый оставшийся сектор должен соответствовать файлу (при инициализации фс все файлы созданы). Все файлы одинакового размера и могут занимать от 1 до M секторов. Данную ФС должно быть можно смонтировать в `/mnt`. Необходимо реализовать регистрацию в VFS для отображения файлов в userspace. В файлы можно писать и читать из них, остальные операции не предусмотрены (переименование, перемещение, статистика и т.д.).

У Superblock-а обязательно должна быть копия. Сектор для размещения копии superblock-а задается параметрами модуля. Необходимо предусмотреть проверку целостности данных superblock-а (подсчёт хеш суммы).

Также необходимо сделать userspace программу (любой удобный язык программирования) для демонстрации корректности работы модуля:
Программа обходит весь диск и в каждый файл сначала пишет случайное число, а затем читает его. С помощью CLI программа позволяет вызывать IOCTL из задания ниже.

### Аргументы модуля:
* Название диска на базе которого должна быть создана ФС
* Смещения N для superblock-ов (отдельно для первой и второй копии)
* Максимальная длина имени файла
* Максимальный размер файла в секторах (M)

### IOCTL:
* Обнуление всех файлов
* Стереть FS
* Получить список всей метаинформации (хэши) из файлов
* Получить маппинг секторов для заданного файла (передается аргументом)

## Основные структуры модуля
### simplefs_super_block
Основная структура суперблока. Хранит ключевые параметры файловой системы и физически записывается в два разных сектора диска (основной и резервный). Используется для проверки целостности при монтировании.
Поля структуры:
- magic - число для строгой быстрой проверки соответствия формата ФС.
- version - текущая версия структуры ФС.
- hash - контрольная сумма полей суперблока. Позволяет ядру понять, не поврежден ли суперблок.
- files_count - фиксированное число файлов в ФС, созданных при форматировании.
- block_size - размер одного блока/сектора устройства (по умолчанию 4096 байт).
- max_file_sectors - максимальный размер файла в секторах.
### simplefs_mapping
Структура, предназначенная для безопасной передачи данных между пространством ядра и userspace через IOCTL.
Поля структуры:
- filename - строковое имя файла для поиска.
- start_sector - номер сектора, с которого физически начинаются данные запрашиваемого файла на диске.
- sector_count - длина файла в секторах.
## Набор команд для демонстрации работоспособности модуля
Дальнейшие команды предполагают использование привилегий суперпользователя (sudo).

### Сборка модуля ядра и userspace-программ:

```
make clean
make
gcc mkfs_simplefs.c -o mkfs_simplefs
gcc test_app.c -o test_app
```
### Создание образа диска и форматирование (инициализация ФС):
Создаем чистый бинарный образ диска размером в 2048 блоков, форматируем его утилитой mkfs_simplefs (она инициализирует метаданные и рассчитывает хэши) и связываем с петлевым интерфейсом loop0
```
sudo umount /mnt/simplefs 2>/dev/null
sudo rmmod simplefs 2>/dev/null
sudo losetup -d /dev/loop0 2>/dev/null

dd if=/dev/zero of=simplefs.img bs=4096 count=2048
./mkfs_simplefs simplefs.img
sudo losetup /dev/loop0 simplefs.img
```
### Загрузка модуля в ядро:

```
sudo insmod simplefs.ko disk_name="/dev/loop0" sb_first_offset=0 sb_second_offset=1024 max_name_len=32 max_file_sectors=4
```
### Монтирование файловой системы:

```
sudo mkdir -p /mnt/simplefs
sudo mount -t simplefs /dev/loop0 /mnt/simplefs
```
## Для просмотра статуса монтирования:

```
sudo dmesg | tail -n 5
```
### Проверка работы с файлами (чтение/запись):
Запуск теста, проверяющего целостность бинарных данных:
```
sudo ./test_app
```

### Проверка корректности работы конвейера и защиты от зануления размера:
```
echo "Hello SimpleFS" | sudo tee /mnt/simplefs/file0
cat /mnt/simplefs/file0
```
### Получение маппинга секторов для файла:

```
sudo ./test_app --map file3
```
### Получение списка хэша:

```
sudo ./test_app --hashes
```
### Обнуление файлов:
Команда сбрасывает в ноль содержимое секторов, выделенных под данные файлов.
```
sudo ./test_app --zero
```
### Стирание файловой системы:
Деструктивная операция, уничтожающая обе копии суперблоков на устройстве.

```
sudo ./test_app --erase
```
## Демонстрация отказоустойчивости и работы второго суперблока:
Для подтверждения корректности работы резервного суперблока выполните следующую последовательность симуляции сбоя:
Пересоздайте чистый образ, отформатируйте его и примонтируйте заново (для чистоты эксперимента):
```
sudo umount /mnt/simplefs 2>/dev/null
dd if=/dev/zero of=simplefs.img bs=4096 count=2048
./mkfs_simplefs simplefs.img
sudo mount -t simplefs /dev/loop0 /mnt/simplefs
```
Запишите контрольные текстовые данные в любой файл, чтобы убедиться в его сохранности:

```
echo "Проверка работы резервной копии" | sudo tee /mnt/simplefs/file3
```
Размонтируйте файловую систему, чтобы корректно затереть сектор на диске:
```
sudo umount /mnt/simplefs
```
Намеренно уничтожьте только первый сектор устройства (где находится исключительно основной суперблок), заполнив его нулями:
```
sudo dd if=/dev/zero of=/dev/loop0 bs=4096 count=1 conv=notrunc
```
Попробуйте примонтировать файловую систему повторно:
```
sudo mount -t simplefs /dev/loop0 /mnt/simplefs
```
Если проверить лог ядра командой ``` sudo dmesg | tail -n 5 ```, можно увидеть, что драйвер зафиксировал повреждение основного суперблока и прозрачно переключился на чтение резервного
``` cat /mnt/simplefs/file3 ```
